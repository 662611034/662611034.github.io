自作ルーター作ろうとして諦めた

2026-01-24

まとめ

自作ルーターを作ってみようと頑張ったが、確定申告などで不具合が出そうだったので諦めた。

発端から計画

特に深い理由はないが、家に自作ルーターがあると楽しそうと思ったため、自作ルーターの作り方を調べた。 最悪ラズパイでも作ることができるし、1Gbpsの速度であればN100程度のCPUで計算能力は足りるとのこと。LANポートだけはWANとLAN用で2つ必要で、デュアルLAN搭載のミニPCで作ることにした。LANポートも足りないし無線APとしてはハードウェアが貧弱で使えないが、そこは既存ルーターをAP兼L2スイッチとしてぶら下げて使えばいいと考えた。

次にソフトの方は、巷ではLinuxベースのOpenWrtとFreeBSDベースのOPNsenseがルーター用OSとして知名度が高いとのことで、FreeBSDは触ったことがないからOpenWrtで作ることにした。

立ち上げは成功、MAP-Eで詰まる

OS入れたての状態でも家のルーターの下流で使うとすぐにルーターとして動作するしインターネットもつながるが、ONUに繋げるにはすこし設定が必要で、その周りはAIに丸投げして済ませた。

一通り設定してから導通確認してみたところ、IPv6のアドレスは取得できる、ルーターからインターネットに向けてのpingも通る、しかしLANからインターネットに出られない。理由を調べてみると、今契約しているBiglobe光はIPv6の回線で、その上で「MAP-E」という技術を使ってIPv4の通信を通していて、MAP-Eが海外ではあまり使われておらず、情報が少ない。

VyOSに乗り換え、MAP-E解決

VyOSという、GUIがない代わりにOpenWrtより柔軟な設定ができるルーター用OSがあるとのことで、そっちに乗り換えることにした。公式の設定ガイドやAIを頼りに設定を済ませて、なんとかこちらはLANクライアントからWANに出るところまでこぎつけた。

少し本筋とは離れるが、VyOSのQuick Startのページに不備があるためここに記しておく。公式のQuick StartではファイアウォールのINPUT設定を「基本的にDROP/REJECTして必要なものだけACCEPT」という方針をとっているが、2026年1月24時点で、DHCPのリクエストを受け取る67番ポートを解放する設定がなく、Quick Start通り設定したらクライアントにIPアドレスを割り当てない。LAN側のUDP67番INPUTをACCEPTするルールが必要だ。

途中で、「MAP-Eの設定は正しいはずなのになぜかpingが通らない」という問題で1日ほど使ったが、後から「ルーター本体からのpingを通さないファイアウォール設定になってただけで、クライアントからの通信は通っていた」ことが発覚した。具体的には

  • LANからのpingは通すがルーター本体からのpingは通さないファイアウォール設定になっていた
  • にもかかわらず、まずルーター本体の通信が出てからクライアントのテストをしようと思い、ずっとルーターからpingを飛ばしていた
  • とりあえずは導通テストということでDNSを設定していなかったせいで、クライアントから外に出られることに気づかなかった

といった、「あくまでもテストだから」って省いた設定が悪さをしたということになる。もし自作ルーターを作ろうとしてこの記事を読んでる人がいたら、ファイアウォールはWAN→LAN以外は全部ACCEPTにして、DNSなど、MAP-Eとは直接関係ない設定もしっかりしてからテストするといいと思う。

何がともあれ、これでめでたくルーターがルーターとして機能するようになった。

MAP-Eのポート割り当ての罠

とりあえず動くようになったので、細かい設定に移る前にAIに全体のチェックを通したところ、IPv4の通信周りで少し不備があると指摘された。

私もよく理解していないため大雑把に書くが、BiglobeでIPv4通信のために使っているMAP-Eという方式は、IPv4通信のために使えるポートが限られており、VyOSで「IPv4通信はこのポートを通してNATするように」と指定する必要がある。そしてその「使えるポート」の番号が飛び飛びの値になっているため、一つの設定にまとめることができず

rule 100 { 
  outbound-interface { 
    name tun0 
  } 
  protocol tcp_udp source { 
    address 192.168.0.0/24 
  } 
  translation { 
    address 123.45.67.89 port 5334-5349 
  } 
}
rule 101 { 
  outbound-interface { 
    name tun0 
  } 
  protocol tcp_udp source { 
    address 192.168.0.0/24 
  } 
  translation { 
    address 123.45.67.89 port 6358-6373 
  } 
}

というふうにルールをいくつも設定していた(ポート番号は適当)。ところが、VyOSは通信をする時上から順番にルールを確認するため、仮にこのように設定した場合5334~5349の15ポートしか使われない。そのポートが全部使用中だと、rule 101にしたがって次のポートを使うのではなく通信ができないとのことだ。

解決する方法は

  • IP範囲を分けて、IPアドレスごとにどのポートを使うか割り当てる
  • 動的ポート割り当てツールを使う

とあるが、後者のツールは実際正しく動作するか不明という問題があり、前者の場合も、ブラウザでIPv4サイトのタブを複数あけたりすると、1台のマシンで15ポートを使い切ることもあるため、あまり意味がない。今時GoogleやYoutubeなどほとんどのサイトはIPv6だが、官公庁のサイトが問題だ。例えば確定申告など行政手続きのために官公庁サイトを複数開くと、リダイレクトや認証関係でIPv4ポートを複数使い潰してしまうリスクがある。実際そうなるか否かは試してみなければわからないが、重要度の高いタスクでインターネットに繋がらない不具合が生じかねないのは避けたい。この場合、サイトが繋がらなくなったらルーターを繋げ直す以外方法がなく、頻度は低いものの影響が大きいリスクと判断し、自作ルーター構想は断念することにした。

今後の展望

今回はMAP-Eという通信方式に対応できず諦めることにしたが、日本で使われているIPv4 on IPv6技術のうち、DS-Liteというものが、自作ルーターで対応しやすく、トラブルも出にくいそうだ。AsahiNetなどが対応しており、今のBiglobeの契約の割引期間が終わったら乗り換えて再チャレンジするために、今回購入したミニPCはとっておこうと思う。
